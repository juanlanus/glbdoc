Vision

The purpose is to keep a record of the exams approved by each glober, enforcing them to take the tests in the right sequence. This way the system collects faithful information about the glober's knowledge.

From the standpoint of the globers (whether in the bench or not) the system provides a means to assess a given knowledge level on a specific subject.

From the standpoint of the Staffing staff, the system provides a soft association between globers and skills, through the tags attached to each exam.

From the standpoint of the exams builders, the system provides a friendly UI for creating and mantaining exams.

Exams and courses

An exam is associated with a given knowlede body, not a course. The globers that take the exam are not forced to take the course. For example high-seniority globers can take and approve several exams in a row out of the knowledge they already have.

It doesn't seem possible to build exams to evaluate high-seniority globers. When such globers land in the bench they should be assigned the task of developing and tagging exam questions, sourcing them from StackOverflow or any other source, including themselves.

Notes

  • The system must know whether a glober is in the bench or not.
  • The system's main input is a glober's answers to the questions of an exam.
  • The exammed glober gets the mark immediately.
  • The answers of every exam are stored in the glober's data, together with the mark and the date.
  • The glober can't review the answers of an exam, only the califications and dates.
  • An exam can't be taken twice in a row: there must be a week or so between attempts.
  • The exams have a "landing page" with the link to the exam proper, and links to studying material, be it made in Globant or available in the Internet.
  • The exam launch page prompts the globers to check their identity and to abide by the honor code.
  • The exams builders can find inspiration in the questions and answers in StackOverflow.
  • The questions are tagged by subject and by level.
  • An exam is a unique set of randomly chosen questions that bear specific tags, according to a set of rules like "30 questions tagged jQuery and easy plus 30 tagged JavaScript and easy".
  • There is a "questions database". A question has a title (the question itself), and a set of possible answers. It might also contain a detailed explanation.
  • There are question formats, like choosing the right one from a set of answers (radio buttons), checking the true answers from a set of slightly confussing alternatives (checkboxes), matching each left-hand part with its corresponding right-hand part, entering textual answers (scrutinized with regexps), etc. The format is part of the quetion design.
  • In addition to the format, a question has an evaluation method.
  • The answers of a question might be partially correct, like for example in multiple choice formats, each good choice may have a value (then each wrong answer should carry a negative value). Some answers should have zero value.
